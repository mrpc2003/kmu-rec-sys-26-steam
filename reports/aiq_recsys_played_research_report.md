# Competition Plan for KMU RecSys 26 Steam Played Prediction

## 1. Problem framing and competition-specific decision rule

The KMU RecSys 26 Steam played-prediction task is a known-user, known-item implicit-feedback ranking problem disguised as binary classification. The verified local data are unusually favorable for collaborative filtering: `train.json` has 175,000 Steam review interactions over 6,710 users and 2,437 games, with no duplicate user-game rows; `pairs.csv` has 19,998 candidate rows over 4,737 users and 2,429 games, with zero cold users, zero cold games, and zero candidate pairs already appearing in train. The fields available in train—`userID`, `gameID`, review `text`, `date`, `hours`, `hours_transformed = log2(1+hours)`, `early_access`, and sparse `found_funny`/`compensation`—support both pure collaborative filtering and feature-rich reranking. The date span, 2010-10-15 through 2018-01-05, also makes temporal validation possible.

The central competition fact is that, for each test user, the hidden labels are balanced 1:1 positive to negative. Because the metric is Accuracy, the correct output mechanism is not a global probability threshold. It is a per-user ranking rule: score every candidate pair for a user, sort the user’s candidates by score, and mark exactly the top half as played. This transforms the problem into “rank each user’s candidate list so the likely positives occupy the upper half.” If a user has an even number of candidates, predict `n_user / 2` positives. If an odd count ever appears, use a deterministic policy selected in validation—usually `floor(n/2)` or `ceil(n/2)` depending on the known validation construction; for the provided test structure, first verify actual per-user candidate counts and whether all are even. Ties must also be deterministic: break by calibrated popularity rank, item ID, or an averaged secondary score rather than random ordering, because a single tie-policy change can move many per-user top-half labels.

The public baselines imply two things. A popularity top-half-like baseline at 0.68413 Accuracy shows that the hidden negatives are not random “any unobserved game” negatives; if they were, global popularity alone would often be even stronger, while if they were very hard negatives, popularity might be weaker. The BPR+popularity+per-user-ranking baseline at 0.72194 Accuracy indicates that pairwise collaborative ranking and popularity calibration both matter. Bayesian Personalized Ranking was designed precisely for implicit feedback where only positive observations are available and the objective is to rank observed items above unobserved items [1]. Implicit ALS similarly treats unobserved entries as low-confidence non-preferences and observed play/review events as higher-confidence positives [2]. The practical conclusion is to treat Accuracy as a constrained ranking metric, not as ordinary binary classification.

## 2. Deep EDA to infer generation structure without cheating

Deep EDA should focus on legal structural facts: data distributions, candidate-set geometry, and train/test shift. It should not attempt to acquire labels, scrape Steam, identify test positives externally, or infer labels from public leaderboard probing. The test candidate file itself is allowed data, so it is legitimate to inspect user counts, item counts, candidate multiplicities, and train-derived features for candidate pairs. This is analogous to using competition-provided test covariates in any supervised contest, provided no target labels are obtained or reverse-engineered.

Start with candidate-set geometry. For every test user, compute the number of candidate rows, whether the count is even, the candidate games’ train popularity ranks, candidate games’ co-occurrence similarity to the user’s history, and the user’s train degree. Plot candidate count by user, user degree by candidate count, and item popularity quantiles among candidates. If many users have exactly two candidates, the task resembles pairwise choice; if many have four, six, or more, rank stability and per-user score calibration become more important. Compare the 4,737 test users against all 6,710 train users by historical degree, total log-hours, first/last review date, and text length. Compare the 2,429 test games against all 2,437 train games by popularity, hours, date coverage, and early-access share. Because there are only eight train games absent from test, most item-level models can score almost every candidate using robust item statistics.

Next, analyze popularity mixtures. For each game, compute `pop_count`, log popularity, unique users, total hours, mean/median `hours_transformed`, and recent popularity in several windows, such as interactions after 2016-01-01 or after 2017-01-01. For each user, compute the mean and median popularity rank of previously played games, the share of their history in head/mid/tail popularity buckets, and the entropy of their item-popularity distribution. Popularity calibration research emphasizes that recommendations should align not only with global popularity but also with each user’s historical popularity preference distribution [3][4]. In this competition, that suggests replacing a raw “popular game wins” rule with features such as “candidate popularity percentile minus user’s historical median popularity percentile” and “candidate is in the user’s preferred popularity bucket.”

Temporal EDA is equally important. Plot interactions by month, item launch-era proxies if available from dates, user recency, and item recency. Because the train period ends on 2018-01-05, hidden positives may plausibly represent future or held-out interactions sampled from the same review log. Validation should therefore test whether recent item popularity, user recency, and drift in user preferences matter. For every candidate pair, compute legal train-only temporal features: days since the user’s last review, candidate item popularity before the user’s last review, candidate item growth rate in the final train months, and overlap between candidate game and the user’s recently reviewed games via co-occurrence neighborhoods.

The `hours` and `hours_transformed` fields should be treated as implicit confidence, not as labels for the test pairs. Long playtime can identify stronger user-item affinity and more reliable item neighborhoods. Hu, Koren, and Volinsky’s implicit-feedback ALS formulation explicitly separates binary preference from confidence, often using confidence weights of the form `1 + alpha * r_ui` [2]. For Steam, test at least three matrices: binary interactions, `hours_transformed`, and clipped/log confidence. The sparse `found_funny` and `compensation` fields are unlikely to drive large standalone gains, but their missingness should be measured. They may be useful as user/item aggregate features: users who write funny reviews may be heavy reviewers; compensated reviews or early-access reviews may indicate niche behavior.

Text EDA should begin with coverage, not modeling. Compute review text length, language/script heuristics if visible, per-user concatenated text length, per-item concatenated text length, and whether text availability correlates with hours or item popularity. Review text can improve recommendation when it captures aspects not fully represented in interactions, and recent work continues to explore user-generated reviews as recommendation signals [5]. However, in this task there are no cold items, so text is a secondary reranking signal rather than the backbone.

Allowed sanity checks are valuable and low risk: verify again that train has no duplicate user-game rows, test has no train-overlapping pairs, every test user and item is known, and no test row is duplicated. Also verify that train and test IDs use identical normalization. These checks are not leakage; they prevent formatting and join errors.

## 3. Validation and negative sampling that mimic the hidden test

The validation design should reproduce the competition decision rule exactly: create candidate pairs for validation users, score them, group by user, predict the top half positive, and compute Accuracy. Ranking metrics such as AUC, MAP, or NDCG can be useful diagnostics, but the selection criterion should be per-user top-half Accuracy because the leaderboard uses Accuracy. Data-splitting choices can strongly affect recommender evaluation, and temporal splits answer different questions than random splits [6][7]. Therefore, use a validation portfolio rather than one split.

| Validation split | Construction | Main value | Main risk | Trust signal |
|---|---|---|---|---|
| Random per-user holdout | For users with enough history, hold out one or more observed games as positives; sample equal negatives per user | Fast, stable, many folds | Can leak future popularity and overstate performance | Good for model debugging, not final selection |
| Test-like per-user sampled holdout | Match test user degree and candidate count distribution; hold out positives; sample 1:1 negatives from unobserved games | Closest to top-half Accuracy | Depends heavily on negative sampler | Primary local metric if public LB correlates |
| Global temporal holdout | Train on older interactions, positives from later interactions; sample negatives per user | Tests future generalization and popularity drift | May underrepresent users/items with older-only activity | Important robustness metric |
| Temporal stress split | Use only the latest months as positives, perhaps post-2017 | Detects recency overfitting and drift | Smaller and noisier | Use as stop-loss check |
| Blended CV | Average ranks or scores across several split types | Reduces public/private mismatch | More compute and bookkeeping | Best for submission decisions |

Negative sampling must be separated into training negatives and validation negatives. Training negatives for BPR, FM, LightGCN, and sampled neural models are optimization devices. Validation negatives define the benchmark. Uniform unobserved negatives are easy; popularity-biased negatives are harder and often closer to recommender test construction; in-community negatives are drawn from items popular among users similar to the target user; hard negatives are high-scoring unobserved items under a baseline. Recent negative-sampling research emphasizes that negative choice materially affects implicit-feedback learning and evaluation reliability [8]. For this competition, build at least four validation sets: uniform 1:1, popularity-matched 1:1, user-community 1:1, and mixed 50% popularity + 25% uniform + 25% hard. Track which split best correlates with public submissions.

A practical recipe is: for each validation user, hold out one or more positives; remove them from the training matrix; sample the same number of negatives from games not observed by that user in the full training data. To mimic the no-overlap test condition, never allow a validation negative that is actually a known positive for that user in the original full data. To mimic known-user/known-item constraints, restrict validation to users and items still present after the split. Then score all candidate pairs and enforce top-half. Store the candidate file once per split to ensure all models are evaluated on identical rows.

Public/private mismatch should be managed like a Kaggle risk, not solved by leaderboard chasing. Kaggle practitioners repeatedly warn that public leaderboard rank can be misleading when the public subset is small or distributionally different from the private subset [9][10]. Keep a table of every submission with local CV scores on all validation splits, public score, model family, and one-line change. If public improvements are not consistent with local improvements, reduce submission frequency and trust the validation blend. Do not tune dozens of threshold or tie-break variants against public feedback, because per-user top-half constraints make it easy to overfit tiny public differences.

## 4. Collaborative filtering model inventory beyond baseline BPR

The strongest first wave should be cheap collaborative models that exploit the small item universe of 2,437 games. Item-item methods are especially attractive: an item-by-item matrix is only about 5.9 million entries dense, and sparse top-k variants are tiny. Amazon’s classic item-to-item collaborative filtering paper is still relevant because it emphasizes precomputing item similarities and scoring users by similarity to their history [11]. For each candidate pair `(u, i)`, compute features such as sum, max, mean, and recency-weighted sum of similarities between candidate game `i` and all games in user `u`’s training history. Use cosine, Jaccard, log-likelihood, BM25-weighted cosine, and `hours_transformed`-weighted cosine. These models are deterministic, fast, and often blend well with BPR.

EASE should be a top-priority experiment. EASE is an “embarrassingly shallow” linear autoencoder for sparse recommendation that learns an item-item weight matrix in closed form from the item Gram matrix with L2 regularization and a zero diagonal [12]. With only 2,437 games, the matrix inversion is feasible on ordinary hardware. Train EASE on binary interactions, log-hour weights, and BM25/TF-IDF-normalized user-item matrices. Sweep regularization broadly, for example `lambda = 50, 100, 300, 1000, 3000`, because the optimal value depends on matrix scaling. Candidate features should include raw EASE score, rank within user candidates, and blended EASE+popularity score.

SLIM is a related but sparse learned item-item model that optimizes a sparse linear reconstruction with nonzero off-diagonal item weights [13]. It is usually slower than EASE but can be more selective because L1 regularization produces sparse neighborhoods. On this data size, a coordinate-descent or ElasticNet-style SLIM is plausible if time permits. Start with a binary matrix, enforce nonnegative weights if the implementation supports it, and tune L1/L2 for sparse but not empty neighborhoods. SLIM is most valuable if EASE over-smooths popular items.

BPR remains central but should not be used only in its baseline form. The BPR paper frames implicit recommendation as pairwise ordering of observed over unobserved interactions [1]. Tune embedding dimension, regularization, learning rate, number of epochs, negative sampler, and item/user bias terms. Use at least binary and confidence-weighted versions; even if BPR’s loss is pairwise, sampling positives proportional to `1 + clipped(hours_transformed)` can emphasize stronger preferences. Try negative sampling distributions: uniform, popularity^0.5, popularity^0.75, and in-batch hard negatives. The validation portfolio should decide whether hard negatives help or simply overfit.

Implicit ALS is a complementary model because it uses all unobserved entries with low confidence rather than only sampled negative pairs. The standard formulation assigns observed interactions positive preference and confidence increasing with interaction strength [2]. For Steam, use `p_ui = 1` if reviewed/played, and `c_ui = 1 + alpha * r_ui`, where `r_ui` can be binary, clipped hours, or `hours_transformed`. Sweep rank 32, 64, 128; alpha 5, 20, 40, 80; regularization 0.01, 0.05, 0.1; and iterations 15-50. ALS scores are often well calibrated for popularity-heavy recommendation and blend well with item-item scores.

Graph methods are high-cost but plausible. LightGCN simplifies graph convolution for recommendation by removing feature transformations and nonlinearities, leaving normalized neighborhood propagation over the user-item graph [14]. NGCF and graph recommender surveys emphasize higher-order collaborative signals on bipartite graphs but at greater tuning and compute cost [15]. In this competition, LightGCN is the preferred neural graph bet: build the bipartite graph from train interactions, train with BPR loss, use 2-4 propagation layers, and score candidate pairs by dot product of final embeddings. Random-walk methods are a cheaper graph alternative: random walks with restart or PersonalRank-style scores can estimate proximity between a user node and candidate item node [16]. Graph scores are especially useful as ensemble diversity if EASE/BPR/ALS disagree.

| Method | Expected benefit | Cost | First settings | Main validation risk |
|---|---:|---:|---|---|
| Calibrated popularity | High baseline, strong tie-breaker | Very low | log item count, recent count, user popularity preference | Overpredicts head games |
| Item-item cosine/co-occurrence | High | Low | binary + log-hours, top-k 100-300 | Favors broad popular neighborhoods |
| EASE | Very high | Low/medium | binary/BM25, lambda 100-3000 | Sensitive to scaling and regularization |
| SLIM | Medium/high | Medium | ElasticNet, sparse top-k | Slower; may overfit small validation |
| BPR variants | High | Medium | dim 64-128, tuned negatives | Negative sampler mismatch |
| Implicit ALS | Medium/high | Low/medium | rank 64-128, alpha 20-80 | Confidence scaling mismatch |
| LightGCN | Medium/high | High | 2-3 layers, BPR loss | Overfitting and tuning time |
| Random walk | Medium | Low/medium | restart 0.15-0.3 | Popularity bias and convergence |

## 5. Feature-rich supervised and hybrid models

Feature-rich models should be rerankers over candidate pairs, not replacements for collaborative filtering. Factorization Machines model pairwise feature interactions and are well suited to sparse categorical recommender features such as user ID, item ID, and side attributes [17]. One-class field-aware factorization machines extend this idea to implicit one-class recommendation settings [18]. For validation candidate pairs, train an FM or field-aware model with labels from held-out positives and sampled negatives. Features should include `userID`, `gameID`, user aggregates, item aggregates, date aggregates, early-access flags, sparse funny/compensation aggregates, and base recommender scores from popularity, item-item, EASE, BPR, ALS, and LightGCN.

A simpler and often stronger approach is a tree or linear meta-model. Build an out-of-fold training table from validation splits where each row is a candidate pair and columns are model scores/ranks plus engineered features. Train logistic regression, ridge regression, LightGBM/CatBoost if allowed, or a shallow FM. Use group-aware validation by user and always evaluate after per-user top-half enforcement. Important features include each base model’s raw score, per-user rank, percentile rank, z-scored score within user candidates, item popularity rank, candidate popularity bucket, user historical popularity bucket, item-item max/sum similarity, candidate recency popularity, and user degree. In final prediction, the meta-model score is still converted to labels by per-user top-half ranking.

Sequence and recency features are cheap and likely valuable. For each user, order reviews by date and compute: last reviewed game, last five games, recency-weighted item-item similarity to candidate, candidate similarity to the most recent game, days since user’s last review, user activity trend, and whether the candidate item was popular in the user’s recent active window. Because Steam preferences may drift by genre, franchise, or multiplayer trend, recency-weighted co-occurrence can outperform all-history averages. Temporal validation is essential: recency features can look excellent on random splits and fail on a different temporal sampling scheme.

Review text embeddings are a later-stage experiment. Public pretrained language models can be used without scraping external labels, as long as the only task-specific text is from provided train reviews. BERT-style embeddings have been explored for recommendation, including multi-BERT embedding approaches [19], and user-generated reviews are a recognized source of preference signals [5]. Construct item profiles by averaging embeddings of reviews for that game and user profiles by averaging embeddings of the user’s reviews, optionally recency-weighted. Candidate features include cosine similarity between user profile and item profile, similarity to the user’s recent-review profile, item text cluster, user text cluster, and text length aggregates. Do not spend early submissions on text unless collaborative models have plateaued, because no cold items/games exist in test.

Hours can be used several ways. In matrix models, they are confidence weights. In supervised rerankers, use user total hours, user median log-hours, item mean hours, item long-play share, and candidate item’s match to user’s typical playtime intensity. A user who mostly reviews 100-hour games may be more likely to play deep strategy/RPG titles than short casual titles, even after popularity adjustment. Since `hours` is observed only for train interactions, candidate features must be item/user aggregates, not pair-specific future hours.

## 6. Ensembling, rank aggregation, and per-user thresholding

The final system should be an ensemble because the candidate label rule is rank-based and heterogeneous models make different errors. Use three representations of every base model: raw score, within-user normalized score, and within-user rank. Raw scores capture magnitude but are not comparable across models; ranks are robust but discard distance; z-scores or percentile scores often work best for blending. Weighted averaging is a common baseline ensemble strategy in Kaggle recommendation notebooks [20], but weights should be selected by local CV, not public leaderboard micro-tuning.

Rank aggregation is particularly useful when base models have different scales. Reciprocal Rank Fusion (RRF) scores an item by summing `1 / (k + rank_m)` across models and is widely used to combine retrieval systems; Kaggle examples also discuss combining retrievers with RRF-like methods [21]. Borda-style rank averaging is even simpler: average candidate ranks across models within each user. For this task, try at least: mean normalized score, weighted normalized score, mean rank, median rank, RRF over EASE/BPR/ALS/item-item/popularity, and a meta-model over ranks. Enforce the top-half constraint only after the final ensemble score.

Popularity calibration should be treated as both a feature and a reranking constraint. Research on mitigating popularity bias and aligning recommendation lists with user popularity preferences supports the idea that users differ in their tolerance for head items [3][4]. For each user, compute historical head-item share using item popularity quantiles. During reranking, penalize candidates whose popularity bucket would move the user’s predicted top half too far from their historical distribution. This should be soft, not hard: the competition rewards Accuracy, not diversity. A practical formula is `final = collaborative_score + w_pop * popularity_score - w_mismatch * abs(candidate_pop_quantile - user_median_pop_quantile)`, with weights selected by CV.

Top-half enforcement is the last step and must be deterministic. The pipeline should output, for each user, a sorted candidate table with columns: final score, all base ranks, tie-break score, predicted label, and position. If candidate count is `m`, mark the first `m/2` rows positive. For validation, compute both ordinary row Accuracy and per-user mean Accuracy. If they differ materially, the public/private score may be sensitive to users with many candidates, so add a group-balanced metric to model selection.

## 7. Leakage, rule compliance, and failure modes

The most important leakage pitfall is confusing allowed structural use of `pairs.csv` with prohibited label inference. It is allowed to know that test users/items are known, that pairs do not overlap train, and that every user is balanced 1:1. It is not allowed to collect Steam answers, scrape current Steam user-game histories, search for hidden labels, use external databases that identify whether the anonymized user played the game, or use leaderboard probing to reverse-engineer labels. Public pretrained language models are acceptable because they do not contain competition-specific labels; external Steam label acquisition is not.

Validation leakage can be subtler. If validation negatives are sampled only from items not observed in the user’s training fold, but some are actually positives later in the full dataset, the validation labels become wrong. For offline validation, sample negatives from items never observed by that user in the full local train file. Conversely, do not remove future data incorrectly when testing temporal generalization: item popularity features in a temporal split must be computed only from the training period, or the model leaks future popularity. Data splitting can make or break recommender evaluation because different split strategies measure different deployment assumptions [6].

Text leakage is also possible. If text embeddings are trained or aggregated using reviews from held-out validation positives, the validation score is inflated. In each fold, item text profiles should be built from fold-train interactions only. User text profiles must exclude held-out positives. For final training on all provided train data, using all train review text is fine because those texts are provided data. Do not augment item text with external Steam store descriptions unless rules explicitly allow that; the prompt restricts work to provided data and public pretrained models.

Per-user top-half ranking creates its own failure modes. A model with better global AUC can lose Accuracy if its within-user ordering is worse for users with many candidates. A model can also look good because it learns the validation negative sampler, not because it identifies positives. Hard-negative validation can over-penalize popularity; uniform validation can over-reward it. Therefore, require improvements to appear across at least two validation regimes before submitting. Tie handling should be frozen early. If many candidates receive identical scores from popularity or item-item models, unstable sorting can cause non-reproducible submissions.

Leaderboard overfitting is a real risk. Public LB feedback is a sample, and Kaggle shakeups occur when competitors optimize to public noise [9][10]. Limit submissions to models that pass prewritten criteria: improve blended CV by a meaningful margin; do not regress badly on temporal stress; and add genuine model diversity or a principled calibration change. Avoid repeated submissions that only adjust one blend weight by 0.01.

## 8. Prioritized experiment ladder for limited Kaggle submissions

**Phase 0: Reproducible scaffold and diagnostics.** Build one canonical data loader, ID maps, sparse matrices, candidate scorer, per-user top-half converter, and validation generator. Reproduce the popularity baseline and the BPR+popularity+per-user-ranking baseline as closely as possible. Produce EDA plots for candidate counts, user/item degree, item popularity in test, temporal distributions, and user popularity preferences. No submission should be made until the offline metric applies the exact top-half rule.

**Phase 1: Cheap high-return models.** Implement calibrated popularity, item-item cosine/co-occurrence, BM25-weighted item-item, and EASE. Submit only after at least one validation split beats the reproduced BPR+popularity baseline and no split collapses. Expected best early candidate: `0.25 popularity + 0.35 EASE + 0.25 item-item + 0.15 BPR`, with per-user normalization before blending. Also test EASE-only plus popularity calibration, because EASE often outperforms more complex models on small implicit matrices [12].

**Phase 2: Tune factor models.** Sweep BPR and implicit ALS with binary and log-hour confidence matrices. Add user/item bias terms where available. Tune negative sampling for BPR and confidence scaling for ALS. Blend the best BPR and ALS with EASE/item-item rather than selecting a single standalone factor model. Submission trigger: blended CV improvement over Phase 1 on test-like sampled holdout and no large temporal-stress regression.

**Phase 3: Supervised reranking.** Create out-of-fold candidate tables and train logistic/ridge/FM/GBM rerankers over base scores and engineered features. Ensure folds are clean and all base-model features are out-of-fold for training rows. Start with linear/ridge or logistic stacking because it is less likely to overfit than a deep model. Add FM if categorical interactions between users, games, and popularity buckets appear valuable [17][18]. Submission trigger: reranker improves per-user top-half Accuracy across at least two validation generators.

**Phase 4: Recency and text.** Add recency-weighted item-item scores, recent popularity, user activity trend, and text embedding similarities from public pretrained models. Treat text as ensemble diversity, not a primary model. If text improves only random CV but not temporal/test-like CV, drop it. If recency improves temporal CV but hurts random CV, include it with a small weight and monitor public correlation.

**Phase 5: Graph models.** Train LightGCN with 2-3 propagation layers and BPR loss; optionally add random-walk proximity. Graph neural methods are justified by their ability to propagate higher-order collaborative signals [14][15], but they cost more tuning time. Submit graph blends only if they add diversity to EASE/BPR/ALS and improve blended CV. NGCF should be attempted only if LightGCN is clearly promising and compute remains.

**Submission selection rules.** Maintain a submission ledger. Spend early submissions on qualitatively different systems: popularity/BPR reproduction, EASE blend, ALS+BPR+EASE blend, supervised stack, recency/text blend, graph blend. Do not submit every hyperparameter sweep. Stop trusting a validation split if it repeatedly predicts public movement in the wrong direction; update the blended CV weights, but do not tune directly to public labels. The final private-LB candidate should be selected by robust CV rank, not necessarily the single best public score.

## 9. Practical implementation checklist

1. Verify all local facts: row counts, user/item counts, no duplicates, no cold users/items, no train-test overlap, per-user test candidate counts.
2. Implement `predict_top_half(scores, userID)` once and reuse it for validation and submission.
3. Build validation files with fixed candidate rows and 1:1 positives/negatives per user.
4. Train and cache base scores: popularity, recent popularity, item-item cosine/Jaccard/BM25, EASE, BPR, ALS.
5. Convert every base score into within-user rank, percentile, and z-score.
6. Blend by CV, then enforce top-half.
7. Add supervised reranking only with out-of-fold features.
8. Add recency/text/graph signals only after cheap CF models plateau.
9. Freeze tie-breakers and submission formatting.
10. Keep rule compliance explicit: no scraping, no label acquisition, no leaderboard probing for labels.

## 10. Concrete feature dictionary and scoring formulas

A competition implementation should make the pair table the central artifact. Each row is a `(userID, gameID)` candidate, and every model contributes one or more columns. The strongest columns are usually not exotic. They are reliable, fold-safe transformations of interaction counts, co-occurrence, latent-factor scores, and within-user ranks. For item popularity, include raw count, `log1p(count)`, item user count percentile, item total `hours_transformed`, item mean and median `hours_transformed`, recent count, and recent-to-all count ratio. For user history, include user degree, user total and median hours, user first and last review date, user activity span, user head-item share, and user median item-popularity percentile. For pairwise collaborative signals, include item-item sum similarity, maximum similarity, mean of top five similarities to the user’s history, recency-weighted similarity, EASE score, SLIM score, ALS dot product, BPR dot product, LightGCN dot product, and random-walk proximity.

Within-user normalization is essential. A raw BPR score for a highly active user may not be comparable to a raw BPR score for a sparse user, but the competition decision is within each user. For every score column `s`, add `rank_s_within_user`, `pct_rank_s_within_user`, `z_s_within_user = (s - mean_user_s) / std_user_s`, and a robust version using median and median absolute deviation. For models with many tied scores, add tie-break columns such as item popularity, EASE score, and deterministic item ID order. These transformed columns are also useful for stacked models because a logistic or tree model can learn when a base ranker is informative for a particular user-degree regime.

The simplest final formula worth testing is a weighted normalized blend: `score(u,i) = w1*z_pop(u,i) + w2*z_itemitem(u,i) + w3*z_EASE(u,i) + w4*z_BPR(u,i) + w5*z_ALS(u,i) + w6*z_recency(u,i) - w7*pop_mismatch(u,i)`, where `pop_mismatch` is the absolute distance between the candidate item’s popularity percentile and the user’s historical median popularity percentile. The weights should be selected on the validation blend, not on the public leaderboard. A rank-only alternative is `score(u,i) = - average_rank(u,i)` over the selected base models, optionally with RRF. Rank-based blends are robust when score scales drift across folds.

For reproducibility, every model should write a candidate-score file with the same row order as `pairs.csv`. The final ensembling notebook should only merge score files, normalize by user, rank by user, and write the submission. This separation makes it easier to diagnose accidental row-order bugs, fold leakage, or inconsistent ID mappings. The most common practical failure in classroom recommender competitions is not a weak model but a silent mismatch between matrix indices, raw IDs, and submission rows.

## 11. Expected ablation sequence and interpretation

Ablations should answer which signal family is carrying the score. Start with global popularity and recent popularity. Add user-calibrated popularity; if it improves, the hidden positives are not merely head items but head items conditional on user taste. Add item-item co-occurrence; if it improves substantially over popularity, candidate positives share neighborhoods with the user’s existing games. Add EASE; if EASE beats item-item, higher-order linear reconstruction is capturing useful structure beyond direct co-play. Add BPR and ALS; if they improve the blend but not standalone Accuracy, their role is diversity rather than primary ranking. Add recency; if it helps temporal validation but not random validation, keep it at low weight. Add text only if it improves users or items with weak collaborative histories.

A useful diagnostic table should report each model by validation split and by user-degree bucket. For example, split users into degree 1-3, 4-10, 11-30, and 31+. Popularity may dominate sparse users because there is little personal history. Item-item and EASE should improve medium-degree users. BPR, ALS, and LightGCN may help dense users where latent preferences are estimable. If a model helps only dense users but hurts sparse users, use a degree-gated blend: higher EASE/BPR weight for dense users and higher popularity weight for sparse users.

Item-degree buckets matter as well. Head candidates are often easy for popularity and ALS, while tail candidates need item-item neighborhoods, SLIM/EASE weights, or text-derived similarity. If validation shows that many false positives are popular games outside the user’s historical taste, increase popularity-mismatch penalties. If many false negatives are moderately popular games similar to the user’s recent history, increase recency-weighted item-item or EASE weight. These ablations turn the leaderboard problem into a set of controllable error modes rather than an opaque model sweep.

## 12. Recommended final submission strategy

A disciplined final strategy is to keep three candidate submissions ready. The first is the most robust blend by averaged validation rank across all splits. The second is a more temporal/recency-heavy blend if public feedback suggests recent popularity matters. The third is a conservative EASE+BPR+popularity blend with minimal supervised stacking, useful if the supervised stack appears unstable. The final private-oriented choice should prefer the robust blend unless the public leaderboard and multiple local splits strongly agree on another variant.

The best use of limited submissions is diversity. A sequence such as baseline reproduction, EASE blend, EASE+BPR+ALS blend, supervised rank stack, recency/text-enhanced blend, and LightGCN-enhanced blend gives more information than six minor changes to one blend. Each submission should have a clear hypothesis and a precomputed local score table. If public score improves while all local splits worsen, record it but do not chase it. If local test-like validation improves and temporal validation is flat, a submission is justified. If temporal validation collapses, the model is probably learning the wrong negative sampler.

The final system should be simple enough to rerun from scratch. It should train all base models on the full provided train data, compute candidate scores only for the provided candidate pairs, normalize and blend scores within each user, enforce the top-half rule, and write deterministic labels. No step should depend on current Steam data, hidden test labels, manual public-LB probing, or non-reproducible random tie breaks. That combination—strong shallow CF, calibrated popularity, careful validation, and conservative ensembling—is the most practical path beyond the 0.72194 BPR+popularity baseline.
